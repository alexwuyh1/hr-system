# 数据库表优化方案

> **更新日期**: 2026-05-03 — 分析现有表结构融合精简空间，提出合并和删除建议

## 1. 索引优化

当前数据库表缺少必要的索引，频繁查询的字段应添加索引以提升查询性能。

员工表应在状态、组织外键和工号字段添加索引。状态字段用于筛选在职员工列表，组织外键用于按部门查询员工，工号字段虽有唯一约束但显式索引可加速登录和考勤关联查询。考勤表应在工作日期和状态字段添加索引，工作日期用于按日查询考勤记录，状态字段用于统计异常考勤。薪资表应在薪资月份字段添加索引，用于按月份查询和统计。组织表应在类型和父组织字段添加索引，类型用于筛选部门岗位职级，父组织用于构建树形结构。权限表应在角色字段添加索引，用于每次请求的权限校验。

```sql
CREATE INDEX IF NOT EXISTS idx_employees_status ON employees(status);
CREATE INDEX IF NOT EXISTS idx_employees_org ON employees(org_id);
CREATE INDEX IF NOT EXISTS idx_attendance_date ON attendance(work_date);
CREATE INDEX IF NOT EXISTS idx_attendance_status ON attendance(status);
CREATE INDEX IF NOT EXISTS idx_salaries_month ON salaries(salary_month);
CREATE INDEX IF NOT EXISTS idx_organizations_type ON organizations(type);
CREATE INDEX IF NOT EXISTS idx_organizations_parent ON organizations(parent_id);
CREATE INDEX IF NOT EXISTS idx_permissions_role ON permissions(role);
```

## 2. 数据一致性约束

员工表的 department 字段与 org_id 外键存在冗余。department 是组织名称的冗余存储，用于避免关联查询，但可能导致数据不一致。建议在应用层保证两者同步，或在数据库层面添加触发器自动更新。

考勤表缺少员工和日期的联合唯一约束，可能导致同一员工同一天产生多条考勤记录。应添加唯一约束防止重复数据。

```sql
CREATE UNIQUE INDEX IF NOT EXISTS idx_attendance_unique 
  ON attendance(employee_id, work_date);
```

组织表的 type 字段应限制为合法值，避免非法类型数据入库。可使用 CHECK 约束限制为部门、岗位、职级三种类型。

```sql
ALTER TABLE organizations ADD CONSTRAINT chk_org_type 
  CHECK (type IN ('部门', '岗位', '职级'));
```

用户表的 role 字段为字符串类型，未通过外键关联角色表。建议在应用层保证角色名称的合法性，或在数据库层面添加外键约束。

## 3. 字段类型优化

考勤规则表设计为单行表，但数据库层面未限制行数。建议在应用层保证仅存在一条记录，或在数据库层面添加触发器限制插入。

考勤表的 status 字段使用英文值（Normal、Late、Absent），与系统中其他中文状态值不一致。建议统一为中文或统一为英文枚举，保持数据风格一致。

薪资表的 base_salary、bonus、deduction 字段使用 REAL 类型，可能存在精度问题。对于金额字段，建议使用 INTEGER 存储分为单位，或使用 TEXT 存储精确小数，避免浮点数精度丢失。

## 4. 外键约束完善

当前员工表的 manager_id 自引用外键未设置级联行为。当管理员员工被删除时，其下属员工的 manager_id 应设为空而非级联删除。

组织表的 parent_id 外键已设置 ON DELETE SET NULL，符合预期。当父组织被删除时，子组织变为根组织。

考勤表和薪资表的外键已设置 ON DELETE CASCADE，符合预期。当员工被删除时，关联的考勤和薪资记录自动清理。

## 5. 冗余字段处理

员工表的 department 字段是 org_id 关联组织名称的冗余副本。保留该字段的优点是查询员工列表时无需关联组织表，提升查询性能。缺点是需要在员工所属组织变更时同步更新。建议在应用层统一通过服务方法更新，确保数据一致性。

如果未来数据量增长到需要严格消除冗余的程度，可移除 department 字段，通过 JOIN 查询获取组织名称。当前数据规模下保留冗余字段是合理的性能优化策略。

## 6. 表融合精简分析

### 6.1 roles 表可删除

当前 roles 表仅包含 id 和 name 两个字段，是一个纯目录表。permissions 表通过 role 字符串字段直接引用角色名称，并未使用 roles 表的外键关联。users 表的 role 字段同样是字符串类型，也未关联 roles 表。整个系统中 roles 表仅用于权限管理页面的角色列表展示和新增角色时的名称校验。

可以将 roles 表的功能合并到 permissions 表中。permissions 表的 role 字段本身就维护了角色名称集合，通过 DISTINCT 查询即可获取所有角色列表。新增角色时只需检查 permissions 表中是否已存在该角色名称即可。删除角色时直接删除该角色在 permissions 表中的所有权限规则。

精简后的 permissions 表同时承担角色目录和权限规则两项职责，减少一张表的同时不影响功能完整性。

### 6.2 attendance_rules 可转为配置存储

attendance_rules 表仅存储一行全局考勤规则配置，包含上班时间、下班时间、迟到宽容、旷工阈值和加班起算五个字段。对于单行配置数据，使用独立表存在过度设计的嫌疑。

可考虑将配置数据转为键值对形式存储在通用配置表中，或使用 JSON 字段存储。但对于当前系统规模，保留独立表的优点是结构清晰、类型安全、查询简单。如果未来配置项大幅增加，再考虑转为通用配置表。

### 6.3 organizations 表的岗位和职级可拆分

organizations 表将部门、岗位、职级三合一，通过 type 字段区分。但岗位和职级是扁平目录，没有层级关系，parent_id 字段始终为空。只有部门类型使用 parent_id 构建树形结构。

将岗位和职级合并到 organizations 表的收益有限，反而导致表结构不够清晰。岗位和职级作为独立的参考数据目录，保持独立表更符合单一职责原则。但考虑到之前的重构方向是减少表数量，当前设计在可接受范围内。

### 6.4 精简后表结构

精简后的数据库表从 8 张减少到 7 张：

| 表名 | 说明 | 变化 |
|------|------|------|
| users | 用户登录 | 不变 |
| employees | 员工主数据 | 不变 |
| attendance | 考勤记录 | 不变 |
| salaries | 薪资记录 | 不变 |
| attendance_rules | 考勤规则 | 不变 |
| organizations | 组织架构 | 不变 |
| permissions | 角色权限 | 合并 roles 表功能 |

## 7. 优先级汇总

| 优先级 | 优化项 | 具体收益 | 验收指标 |
|--------|--------|----------|----------|
| 高 | 添加常用查询字段索引 | 提升列表查询和统计性能 | 查询响应时间降低 50% 以上 |
| 高 | 考勤表添加联合唯一约束 | 防止重复考勤记录 | 同一员工同一天无法插入两条记录 |
| 中 | 删除 roles 表合并到 permissions | 减少表数量，简化结构 | 权限管理功能正常运行 |
| 中 | 组织表添加 CHECK 约束 | 防止非法类型数据入库 | 插入非部门岗位职级类型时报错 |
| 中 | 薪资字段改为 INTEGER 存储 | 避免浮点数精度丢失 | 金额计算结果精确到分 |
| 低 | 统一 status 字段语言风格 | 提升数据一致性 | 所有状态字段使用同一种语言 |
| 低 | 用户表 role 关联角色表 | 增强数据完整性 | 删除角色时关联用户自动处理 |

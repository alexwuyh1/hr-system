## 🔍 详细约束分析

### 1. 用户唯一约束

**约束定义**:
```sql
CREATE TABLE users (
  username TEXT NOT NULL UNIQUE,
  ...
);
```

**业务含义**: 每个用户名必须唯一，不能重复注册

**合理性分析**: ✅ **完全合理**
- **必要性**: 用户名是登录凭证，必须唯一标识用户
- **完整性**: 配合 `NOT NULL` 确保不能为空
- **场景覆盖**: 
  - ✅ 注册时检查重名
  - ✅ 修改用户名时检查冲突

**潜在问题**: 无

---

### 2. 员工编号唯一约束

**约束定义**:
```sql
CREATE TABLE employees (
  employee_no TEXT NOT NULL UNIQUE,
  ...
);
```

**业务含义**: 每个员工必须有唯一的工号

**合理性分析**: ✅ **完全合理**
- **必要性**: 工号是员工的唯一标识，用于考勤、薪资等关联
- **完整性**: 配合 `NOT NULL` 确保不能为空
- **场景覆盖**:
  - ✅ 入职时分配唯一工号
  - ⚠️ 导入时需处理重号（已修复）

**潜在问题**: 
- ⚠️ **导入功能需要特殊处理**：重复导入时应更新而非报错
- ✅ **已修复**: 使用 `findByEmployeeNo` 检查后更新

### 3. ~~薪资记录唯一约束~~ ❌ **已删除**

**原约束定义**:
```sql
CREATE TABLE salaries (
  employee_id INTEGER NOT NULL,
  salary_month TEXT NOT NULL,
  UNIQUE(employee_id, salary_month),  -- ❌ 已删除
  ...
);
```

**原业务含义**: 每个员工每个月的薪资记录只能有一条

**原合理性分析**: ⚠️ **过严，已删除**
- **局限性**: 
  - ❌ 不支持一个月多次发薪（如基本工资 + 奖金分开）
  - ❌ 不支持薪资调整记录历史

**修改决定**: ❌ **删除唯一约束**
- **修改时间**: 2026-05-01
- **原因**: 支持更灵活的薪资管理
- **效果**: 
  - ✅ 支持基本工资、奖金、补贴等分开记录
  - ✅ 支持多次调薪记录

**新场景支持**:
```sql
-- 现在支持同一员工同一月多条记录
INSERT INTO salaries (employee_id, salary_month, base_salary, bonus, note)
VALUES 
  (1, '2026-05', 10000, 500, 'Base'),   -- 基本工资
  (1, '2026-05', 2000, 0, 'Bonus');     -- 奖金
```

**测试验证**: ✅ 通过
```bash
# 基本工资
POST /api/salaries
{"employeeId":1,"salaryMonth":"2026-05","baseSalary":10000,"bonus":500,"note":"Base"}
→ ID: 1 ✅

# 奖金
POST /api/salaries
{"employeeId":1,"salaryMonth":"2026-05","baseSalary":2000,"bonus":0,"note":"Bonus"}
→ ID: 2 ✅
```

### 10. 外键约束（已添加级联）

**约束定义**:
```sql
-- 员工 → 部门（删除部门时，员工部门置空）
FOREIGN KEY (department_id) 
  REFERENCES departments(id) 
  ON DELETE SET NULL

-- 员工 → 职位（删除职位时，员工职位置空）
FOREIGN KEY (position_id) 
  REFERENCES positions(id) 
  ON DELETE SET NULL

-- 考勤 → 员工（删除员工时，级联删除考勤）
FOREIGN KEY (employee_id) 
  REFERENCES employees(id) 
  ON DELETE CASCADE

-- 薪资 → 员工（删除员工时，级联删除薪资）
FOREIGN KEY (employee_id) 
  REFERENCES employees(id) 
  ON DELETE CASCADE

-- 班次 → 员工（删除员工时，级联删除班次）
FOREIGN KEY (employee_id) 
  REFERENCES employees(id) 
  ON DELETE CASCADE

-- 请假 → 员工（删除员工时，级联删除请假）
FOREIGN KEY (employee_id) 
  REFERENCES employees(id) 
  ON DELETE CASCADE

-- 加班 → 员工（删除员工时，级联删除加班）
FOREIGN KEY (employee_id) 
  REFERENCES employees(id) 
  ON DELETE CASCADE

-- 部门 → 父部门（删除父部门时，子部门父级置空）
FOREIGN KEY (parent_id) 
  REFERENCES departments(id) 
  ON DELETE SET NULL
```

**合理性分析**: ✅ **已完善**
- **必要性**: 保证数据完整性和引用一致性
- **级联策略**:
  - `ON DELETE CASCADE`: 删除主记录时，从记录一起删除（考勤、薪资等）
  - `ON DELETE SET NULL`: 删除主记录时，外键置空（部门、职位等）
- **场景覆盖**:
  - ✅ 删除部门时，员工不会丢失，部门字段置空
  - ✅ 删除员工时，相关考勤、薪资记录自动清理
  - ✅ 删除职位时，员工不会丢失，职位字段置空

**修改时间**: 2026-05-01  
**修改内容**: 添加 `ON DELETE CASCADE` 和 `ON DELETE SET NULL`

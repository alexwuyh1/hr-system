# 后端优化方案

## 一、数据库从 SQLite 迁移至 H2

### 为什么选择 H2

当前使用 SQLite 文件存储，其写入锁为文件级——写入时锁整个库文件，数据量大后性能显著下降。同时 SQLite 的 SQL 方言与标准语法存在差异，JPA 需要专门的 Hibernate SQLite Dialect。

H2 是纯 Java 实现的嵌入式关系型数据库。与本项目场景高度匹配：不需要独立安装数据库服务进程，jar 包随 Spring Boot 项目打包即可，应用启动时自动创建本地文件数据库，停止时自动关闭。H2 支持行级锁，并发写入优于 SQLite。内置 MySQL 兼容模式，与 Spring Boot/JPA 集成深度远超 SQLite。

迁移成本极低：改 pom.xml 加一个依赖，改 properties 改一行 URL 和 dialect，业务代码不动。

### 迁移步骤

**pom.xml 变更：**

```
<!-- 移除 SQLite 驱动 -->
<!-- <groupId>org.xerial</groupId>
     <artifactId>sqlite-jdbc</artifactId> -->

<!-- 移除 SQLite Hibernate Dialect -->
<!-- <groupId>org.hibernate</groupId>
     <artifactId>hibernate-community-dialects</artifactId> -->

<!-- 新增 H2 -->
<dependency>
    <groupId>com.h2database</groupId>
    <artifactId>h2</artifactId>
</dependency>
```

**application.properties 变更：**

```
spring.datasource.url=jdbc:h2:file:./data/hr;MODE=MySQL
spring.datasource.driver-class-name=org.h2.Driver
spring.jpa.database-platform=org.hibernate.dialect.H2Dialect
spring.h2.console.enabled=true
spring.h2.console.path=/h2-console
```

### 数据迁移

现有 SQLite 数据通过导出 CSV 再导入 H2 完成。开发/测试阶段的种子数据由 `DataInitializer` 写入，因此直接清掉旧库让初始化脚本重建即可。

## 二、异常处理器精简

### 现状

`GlobalExceptionHandler` 中 `EmployeeNotFoundException`、`EmployeeNotActiveException`、`ResourceNotFoundException`、`DuplicateResourceException`、`InvalidStateException`、`FaceVerificationFailedException`、`InvalidParameterException`、`BusinessException` 八个异常处理方法体完全相同——每段 6 行重复代码。八个方法约 70 行。

这些异常全部实现了 `BusinessException`，都有 `getCode()`、`getMessage()`、`getStatus()` 三个方法。

### 改进：合并为一个 handler

将八个方法替换为一个：

```
@ExceptionHandler(BusinessException.class)
public ResponseEntity<ErrorResponse> handleBusiness(
        BusinessException ex, HttpServletRequest request) {
    ErrorResponse error = new ErrorResponse(
        ex.getCode(), ex.getMessage(), LocalDateTime.now(), request.getRequestURI()
    );
    return new ResponseEntity<>(error, ex.getStatus());
}
```

70 行缩至 7 行。

### 隐藏通用异常的类名

```java
// 改前
"系统内部错误：" + ex.getClass().getSimpleName()
// 改后
"系统内部错误"
```

类名写入日志即可，不暴露给前端。

## 三、JPA 实体关联策略优化

### 现状

`Employee` 实体中 `@ManyToOne(fetch = FetchType.EAGER)` 标记了三个关联字段：orgRef、positionRef、managerRef。每次查询员工列表时 Hibernate 在 SELECT 后自动执行额外的 JOIN/子查询。SERIALIZATION 层已经有 `EmployeeResponse.from()` 做字段映射，控制哪些数据发给前端，但数据库层的 EAGER 加载代价已经付出。

### 改进：改为 LAZY 加载

```java
@ManyToOne(fetch = FetchType.LAZY)
private Organization orgRef;

@ManyToOne(fetch = FetchType.LAZY)
private Organization positionRef;

@ManyToOne(fetch = FetchType.LAZY)
private Employee managerRef;
```

数据量小时差异不明显，但确保架构上可扩展——避免随员工数增长的 N+1 查询。

同理，`Organization` 实体中的 `parent` 和 `grade` 关联也改为 LAZY。

**注意：** EmployeeResponse.from() 中使用了 `employee.getOrgRef()` 等方法获取关联对象。修改为 LAZY 后，这些调用在事务内仍然正常（Hibernate 按需触发一次 SELECT），但在事务外会抛 LazyInitializationException。当前 Controller 默认事务传播到 Service 返回后才关闭，因此无风险。如果有外部调用 Employee.getOrgRef() 的场景，需要加 `@Transactional(readOnly = true)`。

## 四、提取 OrganizationService

### 现状

`OrganizationController` 约 146 行，包含 `buildPositionTree()`（30 行）和 `applyRelations()`（35 行）两大块业务逻辑。Controller 应尽可能薄——只做参数接收和响应返回。

### 改进

新建 `OrganizationService`，将树构建逻辑和关联应用逻辑移入。Controller 只保留路由映射，其余委托给 Service：

```
@GetMapping("/position-tree")
public List<OrganizationNode> positionTree() {
    return organizationService.buildPositionTree();
}
```

Controller 从 146 行降到约 50 行。

## 五、Dashboard 聚合下推到数据库

### 现状

`DashboardService.summary()` 对三张表执行 `findAll()`，全部数据加载到 Java 内存后用 Stream 做过滤、分组、求和。员工数增长到数千人后，单次仪表盘请求会加载三张全表，内存和耗时都会明显增长。

`buildAttendanceIssues()` 先 filter 再 filter——两次 Stream 遍历，而数据库可以一次 WHERE 完成。

### 改进

在 Repository 层加聚合查询方法，让 SQL 做计算：

```java
// AttendanceRepository
@Query("SELECT COUNT(a) FROM Attendance a WHERE a.workDate = :date")
long countByWorkDate(@Param("date") LocalDate date);

// SalaryRepository
@Query("SELECT COALESCE(SUM(s.baseSalary + s.bonus - s.deduction), 0) FROM Salary s")
Double sumTotalPayroll();
```

趋势数据因涉及人工构造月份序列，保持在 Service 层用 Java 计算是合理的。

## 六、EmployeeResponse 去冗余字段

### 现状

`EmployeeResponse` 中 `department` 和 `orgName` 都映射到 `employee.getOrgRef().getName()`，`positionName` 和 `title` 都映射到 `employee.getPositionRef().getName()`。每次序列化发给前端的数据中包含两组完全相同的内容。

### 改进

保留 `orgName` 和 `positionName`，删除 `department` 和 `title`。前端组件同步调整字段引用。

## 七、优化优先级

| 优先级 | 内容 | 收益 | 影响范围 |
|--------|------|------|----------|
| **高** | SQLite 迁移至 H2 | 消除文件锁瓶颈，SQL 标准兼容 | pom + properties |
| **高** | 异常处理器合并 | Controller 层减 60 行 | GlobalExceptionHandler |
| **高** | Employee/Organization 关联改 LAZY | 避免 N+1 查询 | 3 个实体 |
| **中** | 提取 OrganizationService | Controller 减 100 行 | 新增 Service |
| **中** | Dashboard 统计下推到 Repository | 避免全表加载 | 3 个 Repository + 1 个 Service |
| **低** | EmployeeResponse 去冗余字段 | 响应体减小 | 1 个 DTO + 前端组件 |
| **低** | 隐藏通用异常实现类名 | 安全性 | 1 行改动 |

# 项目数据结构重构方案

## 更新内容

分析项目当前数据结构存在的核心问题，提出系统性重构方案。新增权限黑白名单混合模式支持，重构权限管理页面。删除加班功能实现，优化任务优先级排序。

## 问题分析

### 一、员工表单岗位关联问题

当前新增员工时，职位（title）字段为自由文本输入，部门通过下拉框选择部门类型组织。这种设计存在数据不一致风险，员工可以输入任意职位名称，且部门和职位之间没有关联约束。

正确的设计应该是员工关联已存在的岗位，岗位本身已经关联了部门和职级。员工表单中展示岗位选择器，选中岗位后自动带出对应的部门和职级信息，不允许自行选择部门或输入职位名称。员工表中的 `title` 字段应改为引用 `organizations` 表中类型为岗位的记录，通过外键 `position_id` 关联。

### 二、考勤状态自动计算问题

当前 `AttendanceService.create()` 方法允许手动设置考勤状态（Normal、Late、Absent、Leave），状态由用户自行输入而非根据打卡时间和考勤规则自动计算。

考勤状态应该由考勤规则引擎根据上下班时间、迟到宽限分钟数和旷工阈值自动计算得出。打卡时间超过上班时间但未超过旷工阈值的标记为迟到，超过旷工阈值的标记为旷工，正常时间内的标记为正常。手动创建或编辑考勤记录时不应允许设置状态字段，状态由规则引擎在保存后自动计算。

### 三、打卡类型自动判断问题

当前系统通过 `FaceAttendanceService.checkIn()` 和 `FaceAttendanceService.checkOut()` 两个独立接口分别处理签到和签退，需要用户明确选择操作类型。但实际业务中，当日第一次打卡应为签到，已有签到记录的后续打卡应为签退，允许多次打卡。

应该合并为统一的打卡接口，根据当日是否已有签到记录自动判断是签到还是签退。如果当日无签到记录则创建签到，如果当日已有签到但无签退则执行签退，如果当日已有完整签到签退记录则允许再次签到（次日或特殊情况）。

### 四、组织结构树和权限缺失问题

当前存在两个独立的树接口：`/api/organizations/tree` 仅返回部门树，`/api/organizations/position-tree` 返回部门-岗位层级树。前端只需要一个部门-岗位层级树，`/api/organizations/tree` 是冗余接口应删除。

权限系统存在设计缺陷。`JwtAuthenticationFilter` 通过 `PermissionService.isAllowed()` 检查权限，该方法仅支持白名单模式。`DataInitializer` 仅为基础路径写入权限记录，子路径没有对应记录时管理员也无法访问。此外，当前权限模型无法表达黑名单需求，例如人事角色允许访问大部分接口但禁止权限管理，员工角色仅允许人脸打卡。

根本解决方案是引入黑白名单混合模式。权限表新增 `mode` 字段区分 allow 和 deny 规则，检查时先匹配黑名单拒绝，再匹配白名单放行。管理员角色始终全局放行，不受权限表约束。

### 五、Mock 和测试脚本可用性

项目数据结构迭代后，`mock/generate-mock.sh` 和 `test/test-api.sh` 脚本需要适配新的数据结构和 API 接口。员工创建需要关联岗位而非自行输入职位，考勤创建不再需要手动设置状态，组织创建需要指定正确的类型字段。

### 六、权限管理页面重构需求

当前权限管理页面存在多个设计缺陷。角色列表和权限规则分离展示，无法直观查看单个角色的完整权限配置。新增角色时创建一条虚拟权限记录（GET /api/dashboard），角色本身没有独立存储，仅通过权限记录的 role 字段隐式存在。删除角色需要手动遍历删除所有关联权限规则，操作繁琐。

权限表单仅包含角色、方法、路径三个字段，黑白名单混合模式引入后需要增加模式字段。表格没有视觉区分 allow 和 deny 规则，管理员无法快速识别哪些是禁止规则。

需要改为以角色为中心的管理模式。左侧角色列表，点击角色后右侧展示该角色的所有权限规则，支持添加 allow 和 deny 规则。角色管理简化为仅维护角色名称，不绑定虚拟权限。权限规则表格增加模式列，黑名单规则用红色标识，白名单规则用绿色标识。

### 数据库改动

`employees` 表新增 `position_id` 字段，引用 `organizations` 表的 `id`。删除 `title` 和 `department` 冗余字段，这两个字段与 `org_id` 和 `position_id` 外键存储相同信息，保留会导致数据不一致风险。所有使用处改为通过关联对象获取：`orgRef.name` 替代 `department`，`positionRef.name` 替代 `title`。

```sql
ALTER TABLE employees ADD COLUMN position_id INTEGER;
ALTER TABLE employees ADD FOREIGN KEY (position_id) REFERENCES organizations(id) ON DELETE SET NULL;
ALTER TABLE employees DROP COLUMN title;
ALTER TABLE employees DROP COLUMN department;
```

### 后端改动

#### 员工模型和控制器

`Employee` 模型新增 `positionRef` 字段，类型为 `Organization`，通过 `@ManyToOne` 关联岗位。删除 `department` 和 `title` 字段。`EmployeeController` 的员工创建和更新接口中，根据 `positionId` 自动查找岗位，从岗位获取关联的部门和职级。API 响应中通过 `positionRef.name` 和 `orgRef.name` 输出职位名称和部门名称。

```java
@ManyToOne(fetch = FetchType.EAGER)
@JoinColumn(name = "position_id")
private Organization positionRef;

// 响应输出替代
r.department = employee.getOrgRef().getName();
r.title = employee.getPositionRef().getName();
```

#### 考勤状态自动计算

`AttendanceService.create()` 和 `update()` 方法中移除 `status` 字段的接收，保存记录后调用 `AttendanceRuleEngineService` 自动计算状态。`AttendanceRequest` 中移除 `status` 字段。

```java
attendance = attendanceRepository.save(attendance);
attendanceRuleEngineService.computeSingle(attendance);
return attendance;
```

#### 权限黑白名单混合模式

`permissions` 表新增 `mode` 字段，区分 allow（白名单）和 deny（黑名单）两种规则。`PermissionService.isAllowed()` 检查逻辑改为：管理员角色直接放行；非管理员先检查黑名单，匹配则拒绝；再检查白名单，匹配则放行；未匹配任何规则默认拒绝。

```sql
ALTER TABLE permissions ADD COLUMN mode TEXT DEFAULT 'allow';
```

```java
public boolean isAllowed(String role, String method, String path) {
    if ("管理员".equals(role)) return true;
    
    List<Permission> denyRules = permissionRepository.findByRoleAndMethodAndMode(role, method, "deny");
    for (Permission p : denyRules) {
        if (path.startsWith(p.getPathPrefix())) return false;
    }
    
    List<Permission> allowRules = permissionRepository.findByRoleAndMethodAndMode(role, method, "allow");
    for (Permission p : allowRules) {
        if (path.startsWith(p.getPathPrefix())) return true;
    }
    
    return false;
}
```

初始化数据示例：人事角色添加黑名单规则禁止 `/api/permissions`；员工角色添加白名单规则仅允许 `/api/face`。

```java
// 人事 - 黑名单：禁止权限管理
Permission deny = new Permission();
deny.setRole("人事");
deny.setMethod("GET");
deny.setPathPrefix("/api/permissions");
deny.setMode("deny");
permissionRepository.save(deny);

// 员工 - 白名单：仅允许人脸打卡
Permission allow = new Permission();
allow.setRole("员工");
allow.setMethod("POST");
allow.setPathPrefix("/api/face");
allow.setMode("allow");
permissionRepository.save(allow);
```

#### 删除加班相关功能

从考勤规则中删除加班相关字段和计算逻辑。`attendance_rule` 表删除 `overtime_threshold_minutes` 字段，`attendance` 表删除 `overtime_minutes` 字段。`AttendanceRuleEngineService` 中移除加班计算逻辑。前端考勤设置页面移除加班起算分钟数配置项，考勤列表移除加班时长列。

```sql
ALTER TABLE attendance_rule DROP COLUMN overtime_threshold_minutes;
ALTER TABLE attendance DROP COLUMN overtime_minutes;
```

```java
// AttendanceRuleEngineService 中移除加班计算相关代码
```

#### 删除冗余树接口

删除 `OrganizationController` 中的 `/api/organizations/tree` 接口，仅保留 `/api/organizations/position-tree`。前端所有树展示统一使用 position-tree 接口，返回部门-岗位层级结构。

#### 统一打卡接口

新增 `/api/face/attendance` 接口，根据当日考勤记录自动判断签到或签退。无签到记录时创建签到，有签到无签退时执行签退，有完整记录时提示或允许新签到。

```java
Optional<Attendance> todayRecord = attendanceRepository
    .findTopByEmployeeIdAndWorkDateOrderByCheckInDesc(employeeId, today);
if (todayRecord.isEmpty() || todayRecord.get().getCheckIn() == null) {
    return checkIn(employeeId, image);
} else if (todayRecord.get().getCheckOut() == null) {
    return checkOut(employeeId, image);
} else {
    throw new InvalidStateException("考勤", "今日已完成打卡", "次日再打卡");
}
```

#### 权限管理页面重构

权限管理页面改为左右布局。左侧为角色列表，展示除管理员外的所有角色名称（管理员始终拥有所有权限，不纳入管理），点击角色后右侧展示该角色的权限规则表格。新增角色按钮仅创建角色名称，不绑定虚拟权限。

角色列表点击后，右侧加载该角色的所有权限规则。规则表格包含方法、路径、模式三列，模式列显示"允许"或"禁止"，分别用绿色和红色标识。新增权限按钮打开表单，包含方法、路径、模式三个字段，提交后添加到当前选中角色。

删除角色时，先删除该角色的所有权限规则，再从角色列表中移除。删除单条权限规则仅从表格中移除对应行。

### 脚本改动

#### mock/generate-mock.sh

创建员工前先创建岗位，员工创建请求中传入 `positionId` 而非 `department` 和 `title`。考勤创建请求中移除 `status` 字段，仅传入 `employeeId`、`workDate`、`checkIn`、`checkOut`。

#### test/test-api.sh

员工创建测试改用 `positionId` 参数。考勤创建测试移除 `status` 参数。新增组织结构树和角色列表的可用性测试。

## 验收标准

1. 员工表单通过下拉框选择岗位，选中后自动展示关联的部门和职级，不可手动修改
2. 创建考勤记录时不接收 status 参数，状态由规则引擎根据打卡时间和考勤规则自动计算
3. 统一打卡接口根据当日记录自动判断签到或签退
4. 权限黑白名单模式生效，人事禁止访问权限管理，员工仅允许人脸打卡，管理员始终拥有所有权限
5. 权限管理页面左侧角色列表，右侧权限规则，模式列颜色区分允许和禁止，不显示管理员角色
6. 岗位管理页正确显示组织结构树，展示部门与岗位的层级关系
7. 考勤规则和记录中无加班相关字段和计算逻辑
8. mock 脚本成功生成组织、岗位、员工、考勤、薪资数据
9. test 脚本全部测试项通过

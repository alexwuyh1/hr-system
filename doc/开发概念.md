# VibeCoder 必备开发概念

> VibeCoder 不写代码，而是通过自然语言指挥 AI Agent 完成开发。他们不需要精通语法，但必须理解**系统如何运作**，才能有效指导 Agent。

## 🔴 P0 - 核心生存技能

### 1. 分层架构

**定义：** 代码按职责分层，每层做不同的事。

**为什么 VibeCoder 必须懂：**
- 你需要告诉 Agent "在哪一层改"
- 你需要理解 Agent 的修改影响范围
- 你需要判断 Agent 是否改对了地方

**典型分层：**
```
┌─────────────────────────────────────┐
│  前端 (UI)                           │  ← 用户看到的界面
├─────────────────────────────────────┤
│  后端 API (Controller)               │  ← 处理请求、返回数据
├─────────────────────────────────────┤
│  业务逻辑 (Service)                  │  ← 核心业务规则
├─────────────────────────────────────┤
│  数据访问 (Repository/DAO)           │  ← 数据库操作
└─────────────────────────────────────┘
```

**实际场景：**
```
❌ 模糊指令：
"优化一下用户查询"

✅ 精准指令：
"在 Service 层添加缓存，不要改数据库结构"
"在前端添加加载动画，不要改后端 API"
```

**VibeCoder 需要知道：**
- 前端改界面 → 找前端文件
- 后端改逻辑 → 找 Service 层
- 改数据存储 → 找 Repository 层
- 改接口 → 找 Controller 层

---

### 2. 需求拆解

**定义：** 把大目标拆成 Agent 能一步步执行的小任务。

**为什么 VibeCoder 必须懂：**
- Agent 不擅长一次性完成复杂任务
- 拆得越细，Agent 执行越准确
- 每步都能检查，发现偏离及时纠正

**拆解原则：**
```
大目标 → 模块 → 功能 → 步骤 → 具体操作
```

**实际场景：**
```
❌ 大目标：
"做一个用户管理系统"

✅ 拆解后：
第 1 步：创建用户数据表（id, name, email, password）
第 2 步：写用户注册 API（POST /api/users）
第 3 步：写用户登录 API（POST /api/login）
第 4 步：写用户列表 API（GET /api/users）
第 5 步：前端做注册页面
第 6 步：前端做登录页面
第 7 步：前端做用户列表页面
```

**VibeCoder 需要知道：**
- 一个任务只做一件事
- 每步完成后检查效果
- 发现不对立即纠正，不要累积错误

---

### 3. 约束表达

**定义：** 清晰告诉 Agent "做什么"和"不做什么"。

**为什么 VibeCoder 必须懂：**
- Agent 倾向于"多做"（过度实现）
- 不明确的约束 = Agent 自由发挥
- 约束越具体，结果越符合预期

**约束模板：**
```markdown
## 任务
[做什么]

## 约束
✅ 只做：[具体范围]
❌ 不要：[明确禁止]

## 要求
- 代码量：[大约多少]
- 风格：[简单/完整]
```

**实际场景：**
```
❌ 模糊约束：
"写个登录功能"

✅ 明确约束：
"写个登录功能
✅ 只做：验证用户名密码，返回 token
❌ 不要：记住我、邮箱验证、双因素认证
❌ 不要：添加日志、测试、错误处理
代码 50 行以内，简洁优先"
```

**VibeCoder 需要知道：**
- 明确"只做"什么
- 明确"不要"什么
- 明确代码量/复杂度要求

---

### 4. 调试思维

**定义：** 定位问题、描述 bug、验证修复的能力。

**为什么 VibeCoder 必须懂：**
- Agent 写的代码会有 bug
- 你需要准确描述问题，Agent 才能修复
- 你需要验证修复是否有效

**调试步骤：**
```
1. 复现问题（什么操作导致什么错误）
2. 定位范围（前端？后端？数据库？）
3. 描述现象（报错信息、异常行为）
4. 提供上下文（相关代码、操作步骤）
5. 验证修复（修复后是否真的解决）
```

**实际场景：**
```
❌ 模糊描述：
"登录功能有问题"

✅ 精准描述：
"登录功能问题：
- 操作：输入正确用户名密码，点击登录
- 预期：跳转到首页
- 实际：页面报错 '500 Internal Server Error'
- 后端日志：'Cannot read property id of undefined'
- 怀疑：用户查询返回了 null"
```

**VibeCoder 需要知道：**
- 学会看错误信息
- 学会用浏览器开发者工具（F12）
- 学会看后端日志
- 准确描述"预期"和"实际"的差异

## 🟡 P1 - 重要开发技能

### 5. 数据结构基础

**定义：** 理解数据如何组织和存储。

**为什么 VibeCoder 必须懂：**
- 你需要描述数据结构给 Agent
- 你需要理解 Agent 选择的数据结构是否合理
- 你需要判断数据查询方式是否高效

**常见数据结构：**
```
数组/列表：[1, 2, 3, 4, 5]
→ 适合：遍历、按索引访问

对象/字典：{ name: "John", age: 30 }
→ 适合：按字段查找、存储属性

树：部门层级、文件目录
→ 适合：层次结构

图：社交关系、路径规划
→ 适合：网状关系
```

**实际场景：**
```
❌ 模糊描述：
"存储用户信息"

✅ 精准描述：
"用户信息用对象存储：
{
  id: 唯一标识,
  name: 姓名,
  email: 邮箱,
  roles: ["admin", "user"]  // 数组，一个用户多个角色
}"
```

**VibeCoder 需要知道：**
- 数组 vs 对象的使用场景
- 嵌套结构（对象里有数组，数组里有对象）
- 什么数据适合什么结构

### 6. API 设计

**定义：** 前后端交互的接口规范。

**为什么 VibeCoder 必须懂：**
- 你需要告诉 Agent 需要什么 API
- 你需要理解 API 的请求/响应格式
- 你需要判断 API 设计是否合理

**RESTful API 基础：**
```
GET    /api/users          → 获取用户列表
GET    /api/users/123      → 获取单个用户
POST   /api/users          → 创建用户
PUT    /api/users/123      → 更新用户
DELETE /api/users/123      → 删除用户
```

**请求/响应格式：**
```
请求：
POST /api/users
{
  "name": "John",
  "email": "john@example.com"
}

响应：
200 OK
{
  "id": 123,
  "name": "John",
  "email": "john@example.com"
}
```

**实际场景：**
```
❌ 模糊描述：
"做个用户管理的接口"

✅ 精准描述：
"需要以下 API：
- POST /api/users - 创建用户（接收 name, email，返回 id）
- GET /api/users - 获取用户列表（支持分页）
- GET /api/users/:id - 获取单个用户
- DELETE /api/users/:id - 删除用户

所有接口返回格式：{ code: 200, data: {...}, message: "success" }"
```

**VibeCoder 需要知道：**
- HTTP 方法（GET/POST/PUT/DELETE）的用途
- 请求参数放哪（URL 参数 vs 请求体）
- 响应格式统一

### 7. 数据库基础

**定义：** 理解数据如何持久化存储和查询。

**为什么 VibeCoder 必须懂：**
- 你需要描述数据表结构
- 你需要理解查询性能问题
- 你需要判断 Agent 的数据库设计是否合理

**关系型数据库基础：**
```
表：users
┌────┬───────┬──────────────────┬──────────┐
│ id │ name  │ email            │ role_id  │
├────┼───────┼──────────────────┼──────────┤
│ 1  │ John  │ john@example.com │ 1        │
│ 2  │ Jane  │ jane@example.com │ 2        │
└────┴───────┴──────────────────┴──────────┘

表：roles
┌────┬───────┐
│ id │ name  │
├────┼───────┤
│ 1  │ admin │
│ 2  │ user  │
└────┴───────┘

关联：users.role_id → roles.id
```

**常见查询：**
```sql
-- 查询所有用户
SELECT * FROM users;

-- 查询特定用户
SELECT * FROM users WHERE id = 123;

-- 关联查询（用户 + 角色）
SELECT users.name, roles.name 
FROM users 
JOIN roles ON users.role_id = roles.id;
```

**实际场景：**
```
❌ 模糊描述：
"存一下用户数据"

✅ 精准描述：
"创建 users 表：
- id: 主键，自增
- name: 字符串，不能为空
- email: 字符串，唯一，不能为空
- password: 字符串，不能为空
- role_id: 外键，关联 roles 表
- created_at: 时间戳，自动填充

添加索引：email（查询用）"
```

**VibeCoder 需要知道：**
- 表、字段、主键、外键的概念
- 基本查询（SELECT/WHERE/JOIN）
- 索引的作用（加速查询）

### 8. 状态管理

**定义：** 理解应用状态如何变化和管理。

**为什么 VibeCoder 必须懂：**
- 你需要理解前端状态变化
- 你需要描述状态流转逻辑
- 你需要判断状态管理是否合理

**状态概念：**
```
前端状态：
- 用户登录状态（已登录/未登录）
- 页面数据（列表、表单、筛选条件）
- UI 状态（加载中、错误、成功）

后端状态：
- 会话状态（用户身份）
- 缓存状态（热点数据）
- 事务状态（数据一致性）
```

**状态流转：**
```
未登录 → 输入账号密码 → 登录中 → 登录成功 → 已登录
                                    ↓
                               登录失败 → 显示错误
```

**实际场景：**
```
❌ 模糊描述：
"做个登录功能"

✅ 精准描述：
"登录状态流转：
1. 初始状态：未登录，显示登录表单
2. 用户提交：显示加载中，禁用按钮
3. 登录成功：保存 token，跳转到首页
4. 登录失败：显示错误信息，恢复表单

需要处理的状态：
- loading: true/false
- error: null/错误信息
- token: null/字符串"
```

**VibeCoder 需要知道：**
- 状态是什么（应用当前的数据）
- 状态如何变化（触发条件）
- 状态变化后 UI 如何响应

## 🔵 P2 - 进阶技能

### 9. 事务概念

**定义：** 一组要么全部成功、要么全部失败的操作。

**为什么 VibeCoder 需要懂：**
- 你需要识别需要事务的场景
- 你需要告诉 Agent 哪些操作需要原子性
- 你需要理解数据一致性问题

**典型场景：**
```
转账：A 扣钱 + B 加钱
→ 要么都成功，要么都失败
→ 不能只执行一半

创建订单：创建订单 + 扣减库存 + 记录日志
→ 任何一步失败，全部回滚
```

**实际场景：**
```
❌ 不识别事务需求：
"创建订单时扣减库存"

✅ 识别事务需求：
"创建订单需要事务：
1. 创建订单记录
2. 扣减库存
3. 记录操作日志
任何一步失败，全部回滚，保证数据一致"
```

### 10. 安全基础

**定义：** 识别常见安全风险和防御方法。

**为什么 VibeCoder 需要懂：**
- 你需要识别 Agent 代码的安全问题
- 你需要提出安全要求
- 你需要理解基本的安全概念

**常见安全问题：**
```
1. SQL 注入
   ❌ 危险：直接拼接用户输入到 SQL
   ✅ 安全：使用参数化查询

2. XSS 攻击
   ❌ 危险：直接显示用户输入到页面
   ✅ 安全：HTML 转义

3. 密码存储
   ❌ 危险：明文存储密码
   ✅ 安全：哈希加密（BCrypt）

4. 认证授权
   ❌ 危险：不验证用户身份
   ✅ 安全：JWT token 验证
```

**实际场景：**
```
❌ 不提安全要求：
"写个用户登录"

✅ 提出安全要求：
"写个用户登录，要求：
- 密码用 BCrypt 加密存储
- 登录成功返回 JWT token
- 所有 API 接口验证 token
- 用户输入做 SQL 注入防护
- 前端显示用户输入时做 XSS 防护"
```

### 11. 性能意识

**定义：** 识别性能瓶颈和优化方向。

**为什么 VibeCoder 需要懂：**
- 你需要识别慢查询、慢接口
- 你需要告诉 Agent 优化方向
- 你需要理解性能问题的根源

**常见性能问题：**
```
1. N+1 查询
   ❌ 慢：循环查询数据库
   ✅ 快：一次查询所有数据

2. 缺少索引
   ❌ 慢：全表扫描
   ✅ 快：索引查找

3. 重复计算
   ❌ 慢：每次都重新计算
   ✅ 快：缓存结果

4. 大数据传输
   ❌ 慢：一次传 10000 条数据
   ✅ 快：分页传输
```

**实际场景：**
```
❌ 不识别性能问题：
"用户列表加载很慢"

✅ 识别性能问题：
"用户列表加载慢（3 秒），问题：
- 数据库有 10 万用户
- 当前查询没有分页
- 没有添加索引

优化要求：
- 添加分页（每页 20 条）
- 给 name 和 email 字段添加索引
- 只查询需要的字段，不要 SELECT *"
```

### 12. 版本控制

**定义：** 管理代码变更的历史和分支。

**为什么 VibeCoder 需要懂：**
- 你需要理解 Agent 修改了什么
- 你需要回滚错误的修改
- 你需要管理不同版本的代码

**核心概念：**
```
提交（Commit）：保存一次修改
分支（Branch）：独立的开发线
合并（Merge）：把分支的修改合到主线
回滚（Revert）：撤销某次修改
```

**实际场景：**
```
Agent 修改前：
"先创建一个新分支：feature/user-login
在分支上修改，确认没问题后再合并到主分支"

Agent 改坏了：
"回滚到上一次提交，重新开始"
```

**VibeCoder 需要知道：**
- 修改前先创建分支
- 确认没问题再合并
- 改坏了可以回滚

## 🎯 学习建议

### 1. 先掌握 P0 概念
- 这些是 VibeCoder 的**生存技能**
- 不懂这些，无法有效指挥 Agent
- 优先学习，反复练习

### 2. 在实践中学习 P1 概念
- 边用边学
- 遇到问题再深入
- 不需要一次性全部掌握

### 3. P2 概念按需学习
- 遇到相关问题时学习
- 不需要深入，理解概念即可
- Agent 会处理具体实现

### 4. 建立"指令模板"
```markdown
## 任务
[做什么]

## 约束
✅ 只做：[范围]
❌ 不要：[禁止]

## 要求
- [具体要求]

## 流程
1. 先列计划
2. 等待确认
3. 执行
```
